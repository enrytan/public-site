02 / HOW I WORK

I own outcomes, not just tickets. If I built something, I stay responsible for how it holds up in production, not just whether it shipped.

I move fast in unfamiliar territory. Dropping into a codebase or a problem I haven't seen before and getting my bearings quickly is one of my stronger traits.

I try to reduce friction instead of adding to it. When something is fragile, I would rather fix the root cause than stack another workaround on top.

I look for the intent behind a request, what someone is actually trying to solve, so I build the right thing rather than just the literal ask.

I use AI tools heavily but deliberately. I've built a scope driven workflow for AI-assisted coding that keeps the output tight and reviewable instead of letting it sprawl.

It's a large, data heavy application, the kind with big server paginated tables and deeply nested, interdependent forms. I worked mostly on the ad group side, and I spent most of my time in the optimization area, which is the most complex part of the product. This is also where my professional React experience started. Much of the real work was untangling edit mode and replacing fragile patterns so invalid combinations cannot be saved.

  • Owned the ad group optimization flow (optimizer, predictive models, KPIs, bidding), built with React Hook Form and Zod for the interdependent validation and TanStack Query for the many dependent lookups.

  • Built the Next.js API routes, a BFF layer between the React app and the .NET backend, that handle auth, route to the right service, and reshape the data.
  • Managed shared state across the app with Zustand, and shipped changes behind DevCycle feature flags for gradual, controlled rollout.
